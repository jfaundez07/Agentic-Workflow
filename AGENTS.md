# Agentic Workflow

OpenCode agent definitions. **Config/doc repo** — no application code, no build scripts, no tests, no CI, no linter.

## Agent Files

- **`agents/*.md`** — each file has YAML frontmatter (mode, permissions, temperature) that OpenCode reads as executable config. Be careful editing — incorrect frontmatter breaks the agent.
- **No `opencode.json`** in this repo. Config lives entirely in markdown frontmatter.
- **One primary agent** (runs standalone): `tech-lead.md` (orchestrator).
- **Seven sub-agents** (dispatched via `task` tool): analyst, code-documenter, commiter, developer, planner, reviewer, tester.
- **Permission model**:
  - Tech Lead: `bash: true`, `write`/`edit` denied globally; allowed only for `docs/plans/` and `docs/workflow-log.json`.
  - Reviewer: `bash: false`. The only sub-agent without shell access.

## Tech Lead Pipeline

Planner creates `docs/plans/plan-<id>.md` (sequential IDs from 1); subsequent agents read it. Analyst runs independently to produce `docs/analysis.md`. Code Documenter adds inline docs to changed files after Developer completes. Commiter is a fallback, not a pipeline step.

The Tech Lead logs every handoff to `docs/workflow-log.json` (JSON, not markdown) and passes structured JSON context between sub-agents. Always dispatch sub-agents via the `task` tool.

## Runtime Artifacts

- **`.opencode/` is gitignored** (root `.gitignore`). It holds runtime artifacts generated by agents (analysis, plans, etc.).
- **Installation**: `cp -r agents/ ~/.config/opencode/agents && cp -r resources/ ~/.config/opencode/resources`

## Commit Convention

Conventional Commits with scoping. Prefixes: `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`. Always scope to the changed component (e.g., `fix(tester):`, `refactor(agents):`).
